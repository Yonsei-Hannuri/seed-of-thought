## 백엔드 소스코드 작성 원칙
### 레이어 구조
  URLs
    - ViewSet         # HTTP 처리, 권한 체크
    - Serializer      # 요청 검증 / 응답 변환
    - Component       # 외부 서비스 추상화 (S3, LLM, OAuth 등)
    - Job             # 비동기 처리 (Threading)
    - Model           # 데이터 구조 (Django ORM)

  각 레이어는 인접한 레이어하고만 통신한다. ViewSet이 Model을 직접 조작하지 않고 Serializer를 통한다.
### 규칙
  네이밍
  - 클래스: PascalCase (DetgoriViewSet, ObjectStorage, IsOwnerOrReadOnly)
  - 함수/변수: snake_case (get_queryset, perform_create)
  - 내부 전용 함수: 언더스코어 prefix (_create_derived, _parse_date)
  ViewSet
  - HTTP 처리와 권한 체크만 담당
  - 비즈니스 로직은 perform_create / perform_destroy 훅 또는 Job으로 위임
  - 쿼리 필터링은 get_queryset() 오버라이드로 처리
  Component
  - 외부 서비스는 반드시 추상 클래스 뒤에 숨긴다 (ObjectStorage → S3)
  - 모듈 레벨 싱글턴으로 노출 (component/__init__.py에서 인스턴스 생성)
  - 구현체 교체 시 호출부 코드 변경 없어야 함 (OCP)
  Serializer
  - 요청 검증 + 응답 변환 두 가지 역할만
  - 중첩 Serializer로 연관 데이터 구성 (source= 활용)
  - 비즈니스 로직 금지
  Permission
  - 권한 규칙은 반드시 Permission 클래스로 분리
  - ViewSet에 인라인으로 작성하지 않는다
  Job
  - 비동기 사이드 이펙트는 Job 클래스로 분리
  - 내부 로직은 private static 메서드로 작성
  - 시작/종료 상태를 DB 플래그로 관리
### 하지 말 것
  - ViewSet 안에 비즈니스 로직 직접 작성
  - except Exception bare catch
  - Serializer 안에서 외부 서비스 호출
  - 동일한 검증 로직을 ViewSet과 Serializer 양쪽에 중복 작성
